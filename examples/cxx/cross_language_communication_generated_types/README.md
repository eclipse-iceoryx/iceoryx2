# Cross-Language Communication With Generated Types

> [!CAUTION]
> Every payload you transmit with iceoryx2 must be compatible with shared
> memory. Specifically, it must:
>
> * be self contained, no heap, no pointers to external sources
> * have a uniform memory representation, ensuring that shared structs have the
>     same data layout
> * not use pointers to manage their internal structure
> * must be trivially destructible, see `std::is_trivially_destructible`
>
> Data types like `std::string` or `std::vector` will cause undefined behavior
> and may result in segmentation faults. We provide alternative data types
> that are compatible with shared memory. See the
> [complex data type example](../complex_data_types) for guidance on how to
> use them.

This example shows how to make a C++ data type cross-language compatible **when
you cannot modify its definition** - for instance because it is generated by an
IDL/code generator such as rosidl.

It is the non-intrusive counterpart to the
[cross-language communication basics example](../cross_language_communication_basics),
which sets the type name with the `IOX2_TYPE_NAME` member. Here the structs in
[`message_data.hpp`](src/message_data.hpp) carry **no** `IOX2_TYPE_NAME`; instead
the cross-language type name is attached from the outside with
`IOX2_DEFINE_TYPE_NAME` in [`message_data_with_type_name.hpp`](src/message_data_with_type_name.hpp).

Because the resulting type name, layout and service name are identical to the
basics example, the applications here are wire-compatible with the basics
applications in every supported language.

## How to Build

Before proceeding, all dependencies need to be installed. You can find
instructions in the [C++ Examples Readme](../README.md).

When you want to run the C++ publisher and subscriber applications, you first
have to build the C++ examples:

```sh
cmake -S . -B target/ff/cc/build -DBUILD_EXAMPLES=ON
cmake --build target/ff/cc/build
```

## How to Run

To observe the communication in action, open two separate terminals and execute
the following commands:

### Terminal 1

Run the C++ subscriber application:

```sh
./target/ff/cc/build/examples/cxx/cross_language_communication_generated_types/example_cxx_cross_language_communication_generated_types_subscriber
```

### Terminal 2

Run the C++ publisher application:

```sh
./target/ff/cc/build/examples/cxx/cross_language_communication_generated_types/example_cxx_cross_language_communication_generated_types_publisher
```

Feel free to also run the subscriber and publisher applications from the
[cross-language communication basics example](../cross_language_communication_basics)
(in any language) simultaneously - they share the same service and type names,
so they communicate with the applications of this example.

> [!TIP]
> You may hit the maximum supported number of ports when too many publisher or
> subscriber processes are running. Check the [iceoryx2 config](../../../config)
> to set the limits globally or refer to the
> [API of the Service builder](https://docs.rs/iceoryx2/latest/iceoryx2/service/index.html)
> to set them for a single service.

## How to set a type name without modifying the type

To communicate, publisher and subscriber applications must agree on the service
configuration, including the payload and the user header type name. When a type
cannot carry an `IOX2_TYPE_NAME` member, specialize `iox2::TypeNameSpecialization`
for it - most conveniently via the `IOX2_DEFINE_TYPE_NAME` macro at global
scope:

```cxx
// message_data.hpp - the (e.g. generated) types, unchanged
struct TransmissionData {
    std::int32_t x;
    std::int32_t y;
    double funky;
};

struct CustomHeader {
    std::int32_t version;
    std::uint64_t timestamp;
};
```

```cxx
// message_data_with_type_name.hpp - includes the types and attaches their type names
#include "iox2/type_name.hpp"
#include "message_data.hpp"

IOX2_DEFINE_TYPE_NAME(TransmissionData, "TransmissionData");
IOX2_DEFINE_TYPE_NAME(CustomHeader, "CustomHeader");
```

The string can be a literal, or any expression yielding a `const char*` - for
example a generator's own type-name accessor - so the name never has to be
duplicated by hand.

`IOX2_DEFINE_TYPE_NAME` is only a convenience wrapper. If you prefer, you can
specialize `iox2::TypeNameSpecialization` directly - it must be done at global
scope, since an explicit specialization has to live in a namespace enclosing
`iox2`:

```cxx
// message_data_with_type_name.hpp - the macro-free equivalent
#include "iox2/type_name.hpp"
#include "message_data.hpp"

template <>
struct iox2::TypeNameSpecialization<TransmissionData> {
    static auto value() -> const char* {
        return "TransmissionData";
    }
};

template <>
struct iox2::TypeNameSpecialization<CustomHeader> {
    static auto value() -> const char* {
        return "CustomHeader";
    }
};
```

Only the type name is overridden; size, alignment and the type variant remain
derived from the C++ type. When the type names match and the structures share
the same memory layout, the C++ applications and applications written in other
supported languages can communicate.

<!-- markdownlint-disable MD028 -->

> [!NOTE]
> `IOX2_DEFINE_TYPE_NAME` must be invoked at global scope, and a type must not
> define both a `TypeNameSpecialization` and an `IOX2_TYPE_NAME` member. Include
> the header that defines the type name (here `message_data_with_type_name.hpp`)
> wherever the type is used, so that every translation unit agrees on the name.

> [!NOTE]
> For the communication with Rust applications, you don't need to provide a type
> name for `(u)int{8|16|32|64}_t`, `float`, `double` and `bool` payloads. These
> types are automatically translated into the Rust equivalents.
